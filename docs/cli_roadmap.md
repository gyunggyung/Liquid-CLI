# Liquid-CLI 미래 로드맵 (Technical Roadmap & GDPO Strategy)

이 문서는 Liquid-CLI를 세계 최고 수준의 자율형 터미널 에이전트로 진화시키기 위한 기술적 상세 구현 계획과 GDPO 강화학습 전략을 다룹니다.

---

## 🛠️ 1. 차세대 기능 상세 구현 계획

### A. 자율 주행 루프 (Agentic Loop)
- **구현 방식**: `main.py`의 `ask` 함수를 `while` 루프로 감쌉니다.
- **로직**:
  1. 에이전트가 `commands`와 함께 `task_complete: false`를 반환하면 명령어 실행.
  2. 실행 결과(stdout/stderr)를 다음 턴의 프롬프트 입력을 위해 `history`에 누적.
  3. 에이전트가 실행 결과를 보고 "목표 달성"을 판단하여 `task_complete: true`를 보낼 때까지 무한 반복(또는 Max Step 제한).
- **효과**: 사용자가 매번 명령어를 수락하거나 새로 타이핑할 필요 없이, 지시 한 번에 복잡한 작업을 끝까지 완수합니다.

### B. 시스템 인지 로직 (Dynamic Context Injection)
- **구현 방식**: CLI 실행 시 초기에 `uname -a`, `lsb_release -a`, `which git jq curl` 등의 명령어를 비가시적으로 먼저 실행합니다.
- **로직**:
  - 수집된 정보를 시스템 프롬프트의 전처리(Pre-fill) 단계에 삽입합니다.
  - "당신은 현재 Ubuntu 22.04 환경이며, jq와 curl이 설치되어 있습니다"라는 정보를 모델에게 명시적으로 제공하여 헛발질(명령어 오류)을 줄입니다.

### C. 컨텍스트 길이에 따른 속도 최적화 (KV Cache Management)
- **개념**: 터미널 실행 기록이 길어지면 추론 속도가 느려지는 문제를 해결합니다.
- **구현**: 
  - `llama-cpp-python`의 KV Cache 재사용 기능을 활성화합니다.
  - 가변적인 LFM2 아키텍처에 맞춰, 오래된 실행 결과는 요약(Summarization)하거나 슬라이딩 윈도우 방식으로 관리하여 일관된 속도를 유지합니다.

## 🧠 2. GDPO 강화학습 전략 (GDPO Considerations)

모델이 더 짧고 정확하며 효율적인 길을 찾도록 학습시키는 것이 핵심입니다.

### 🎯 핵심 리워드(Reward) 설계
GDPO 과정에서 다음 4가지 요소를 보상 함수로 정의하여 최적화합니다.

1. **포맷 준수 보상 (Format Compliance)**
   - JSON 구조가 조금이라도 깨지면 강한 페널티.
   - 특히 `analysis`, `plan`, `commands` 필드가 필수적으로 포함되어야 함.

2. **명령어 유효성 보상 (Command Validity)**
   - 존재하지 않는 명령어나 오타가 섞인 명령어를 제안하면 페널티.
   - 실제 리눅스 쉘 환경에서 문법적으로 올바른지 검증하는 로직을 리워드에 포함.

3. **효율성 및 길이 제어 보상 (Conciseness & Efficiency)**
   - **가장 중요**: 목표 달성까지 걸리는 명령어의 개수와 출력 텍스트의 총 길이를 최소화할 때 보상 점수 부여.
   - 에이전트가 "설명충"이 되지 않고 바로 실행 가능한 코어로직만 뱉도록 유도함 (추론 속도 향상과 직결).

4. **추론 품질 보상 (Reasoning Quality)**
   - `analysis`와 `plan`의 내용이 실제 `commands`와 논리적으로 연결되는지 평가.
   - 작업과 무관한 `ls`를 남발하지 않고, 최소한의 확인 작업만 거친 후 본론으로 갈 때 가산점.

### ⚠️ GDPO 학습 시 주의점
- **KL Divergence 관리**: 강화학습 도중 모델이 완전히 망가지는 것을 방지하기 위해, SFT 모델과의 차이(KL)를 적절히 유지해야 합니다.
- **MoE/Liquid 특성**: LFM2는 8B 모델임에도 실제 활성 파라미터는 적습니다. 학습 시 안정성을 위해 FP8보다는 BF16 기반의 LoRA 학습을 권장합니다.
- **데이터 다양성**: 단순히 `ls`나 `cd`만 반복하는 데이터가 아닌, 파이프라인(`|`)과 복잡한 필터링(`awk`, `sed`)을 사용해 한 번에 문제를 해결하는 "Goated" 데이터를 리워드 모델에 많이 노출시켜야 합니다.
